DATABASE STRATEGY RECOMMENDATION
═══════════════════════════════════════════════════════════════════════════════

CURRENT: SQLite (instance/wealthsync.db)

ANALYSIS:
SQLite Limitations:
  ❌ File-based storage not cloud-native
  ❌ No true concurrent write support (4-5 concurrent users max)
  ❌ Ephemeral storage on Render (data lost on container restart)
  ❌ No native backups
  ❌ Single point of failure

Render Environment Concerns:
  ❌ Render has ephemeral file systems
  ❌ SQLite file at /instance/wealthsync.db is lost on deploy/restart
  ❌ No persistent volume support on free tier
  ❌ Multi-dyno deployments get separate SQLite instances

RECOMMENDATION:

Option A: SQLite (Beta Phase Only)
  ✓ No setup required
  ✓ Suitable for initial testing with <5 concurrent users
  ✗ Data loss on container restart
  ✗ Not production-ready
  → Use only for MVP/beta with small user base

Option B: PostgreSQL (Recommended for Production) ⭐
  ✓ Cloud-native, fully managed on Render
  ✓ Data persists across deployments
  ✓ Automatic backups by Render
  ✓ Unlimited concurrent connections
  ✓ ACID transactions + data integrity
  ✓ Industry-standard for Flask apps
  ✓ Free tier available on Render
  → Required for public launch

MIGRATION PATH:
1. Start with SQLite for beta testing (if launching small)
2. Before public launch, migrate to PostgreSQL
3. SQLAlchemy makes migration straightforward (just change connection string)
4. Data migration script can be automated

DECISION:
┌────────────────────────────────────────────────────────┐
│ For Production: MUST use PostgreSQL                     │
│ For MVP/Beta: SQLite acceptable (small user count)      │
│ Migration: Easy with SQLAlchemy                         │
└────────────────────────────────────────────────────────┘

KNOWN LIMITATIONS & CONSIDERATIONS

  If deploying with SQLite on Render:
  - Data will be lost when container restarts
  - Suitable only for beta/MVP with <5 concurrent users

  - May require Gmail App Password (not regular password)
  - Test email delivery before going live

OAuth Token Security:
  - Google tokens are exchanged server-side
  - Client-side has no access to sensitive tokens
  - HTTPS enforced for all OAuth flows
